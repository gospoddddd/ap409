Репозиторий: пример структуры для GitLab / Git

## Цель

1. Освоить базовые операции с Git (ветки, коммиты, merge, теги).
2. Разработать несинтезируемый APB master и простой APB slave на SystemVerilog.
3. Выполнить несколько операций чтения/записи по шине APB.

## Структура репозитория

- `README.md` — этот файл, краткое описание проекта.
- `src/apb_master.sv` — несинтезируемый APB master с задачами `apb_write` и `apb_read`.
- `src/apb_slave.sv` — простой APB slave с регистрами и `$display` сообщений.
- `src/apb_top_tb.sv` — верхний уровень / тестбенч, связывающий master и slave.
- `sw/hello_git.py` — простая проверочная программа на Python.

## Рекомендованный порядок коммитов для ЛР

1. **Первый коммит**  
   - Добавить `README.md` и, например, `sw/hello_git.py`.  
   - Сообщение коммита: `init repo with README and hello_git`.

2. **Ветка с обновлением (например, `feature/apb_master`)**  
   - Добавить `src/apb_master.sv`.  
   - Обновить `README.md` (описать master).  
   - Отдельный коммит: `add apb master`.

3. **Добавление APB slave**  
   - В этой же ветке добавить `src/apb_slave.sv`.  
   - Обновить `README.md` (описать slave).  
   - Коммит: `add apb slave`.

4. **Добавление верхнего уровня / тестбенча**  
   - Добавить `src/apb_top_tb.sv`.  
   - Коммит: `add top-level apb testbench`.

5. **Merge ветки в исходную (обычно `main` или `master`)**  
   - При конфликте — вручную решить и сделать merge-коммит.

6. **Финальная версия**  
   - После успешной симуляции сделать финальный коммит.  
   - Поставить тег, например: `v1.0_lab1`.

## Как запускать симуляцию (общая идея)

Ниже общий пример для ModelSim / QuestaSim (имена команд могут отличаться в вашей среде):

```tcl
vlog src/apb_master.sv src/apb_slave.sv src/apb_top_tb.sv
vsim work.apb_top_tb
run -all
```

Или аналогом в используемом симуляторе.  
При удачном запуске в консоль должны печататься `$display` сообщения от slave о записях/чтениях.
